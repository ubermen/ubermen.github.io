90 ### Pull Request Guidelines
91 
92 Before you submit a pull request from your forked repo, check that it
93 meets these guidelines:
94 
95 1. The pull request should include tests, either as doctests, unit tests, or
96 both. The airflow repo uses [Travis CI](https://travis-ci.org/apache/incubator-airflow)
97 to run the tests and [codecov](https://codecov.io/gh/apache/incubator-airflow)
98 to track coverage. You can set up both for free on your fork. It will
99 help you making sure you do not break the build with your PR and that you help
100 increase coverage.
101 2. Please [rebase your fork](http://stackoverflow.com/a/7244456/1110993),
102 squash commits, and resolve all conflicts.
103 3. Every pull request should have an associated
104 [JIRA](https://issues.apache.org/jira/browse/AIRFLOW/?selectedTab=com.atlassian.jira.jira-projects-plugin:summary-panel).
105 The JIRA link should also be contained in the PR description.
106 4. Preface your commit's subject & PR's title with **[AIRFLOW-XXX]**
107 where *XXX* is the JIRA number. We compose release notes (i.e. for Airflow releases) from all commit titles in a release.
108 By placing the JIRA number in the commit title and hence in the release notes,
109 Airflow users can look into JIRA and Github PRs for more details about a particular change.
110 5. Add an [Apache License](http://www.apache.org/legal/src-headers.html)
111  header to all new files
112 6. If the pull request adds functionality, the docs should be updated as part
113 of the same PR. Doc string are often sufficient. Make sure to follow the
114 Sphinx compatible standards.
115 7. The pull request should work for Python 2.7 and 3.4. If you need help
116 writing code that works in both Python 2 and 3, see the documentation at the
117 [Python-Future project](http://python-future.org) (the future package is an
118 Airflow requirement and should be used where possible).
119 8. As Airflow grows as a project, we try to enforce a more consistent
120 style and try to follow the Python community guidelines. We track this
121 using [landscape.io](https://landscape.io/github/apache/incubator-airflow/),
122 which you can setup on your fork as well to check before you submit your
123 PR. We currently enforce most [PEP8](https://www.python.org/dev/peps/pep-0008/)
124 and a few other linting rules. It is usually a good idea to lint locally
125 as well using [flake8](https://flake8.readthedocs.org/en/latest/)
126 using `flake8 airflow tests`
127 9. Please read this excellent [article](http://chris.beams.io/posts/git-commit/) on
128 commit messages and adhere to them. It makes the lives of those who
129 come after you a lot easier.

138 #### Running unit tests
139 
140 We *highly* recommend setting up [Travis CI](https://travis-ci.org/) on
141 your repo to automate this. It is free for open source projects. If for
142 some reason you cannot, you can use the steps below to run tests.
143 
144 Here are loose guidelines on how to get your environment to run the unit tests.
145 We do understand that no one out there can run the full test suite since
146 Airflow is meant to connect to virtually any external system and that you most
147 likely have only a subset of these in your environment. You should run the
148 CoreTests and tests related to things you touched in your PR.
149 
150 To set up a unit test environment, first take a look at `run_unit_tests.sh` and
151 understand that your ``AIRFLOW_CONFIG`` points to an alternate config file
152 while running the tests. You shouldn't have to alter this config file but
153 you may if need be.
154 
155 From that point, you can actually export these same environment variables in
156 your shell, start an Airflow webserver ``airflow webserver -d`` and go and
157 configure your connection. Default connections that are used in the tests
158 should already have been created, you just need to point them to the systems
159 where you want your tests to run.
160 
161 Once your unit test environment is setup, you should be able to simply run
162 ``./run_unit_tests.sh`` at will.
163 
164 For example, in order to just execute the "core" unit tests, run the following:
165 
166 ```
167 ./run_unit_tests.sh tests.core:CoreTest -s --logging-level=DEBUG
168 ```
169 
170 or a single test method:
171 
172 ```
173 ./run_unit_tests.sh tests.core:CoreTest.test_check_operators -s --logging-level=DEBUG
174 ```
175 
176 For more information on how to run a subset of the tests, take a look at the
177 nosetests docs.
178 
179 See also the list of test classes and methods in `tests/core.py`.

181 ### Changing the Metadata Database
182 
183 When developing features the need may arise to persist information to the the
184 metadata database. Airflow has [Alembic](https://bitbucket.org/zzzeek/alembic)
185 built-in to handle all schema changes. Alembic must be installed on your
186 development machine before continuing.
187 
188 ```
189 # starting at the root of the project
190 $ pwd
191 ~/airflow
192 # change to the airflow directory
193 $ cd airflow
194 $ alembic revision -m "add new field to db"
195  Generating
196 ~/airflow/airflow/migrations/versions/12341123_add_new_field_to_db.py
197 ```
