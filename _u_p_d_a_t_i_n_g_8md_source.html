23 ## Airflow 1.8
24 
25 ### Database
26 The database schema needs to be upgraded. Make sure to shutdown Airflow and make a backup of your database. To
27 upgrade the schema issue `airflow upgradedb`.
28 
29 ### Upgrade systemd unit files
30 Systemd unit files have been updated. If you use systemd please make sure to update these.
31 
32 > Please note that the webserver does not detach properly, this will be fixed in a future version.
33 
34 ### Tasks not starting although dependencies are met due to stricter pool checking
35 Airflow 1.7.1 has issues with being able to over subscribe to a pool, ie. more slots could be used than were
36 available. This is fixed in Airflow 1.8.0, but due to past issue jobs may fail to start although their
37 dependencies are met after an upgrade. To workaround either temporarily increase the amount of slots above
38 the the amount of queued tasks or use a new pool.
39 
40 ### Less forgiving scheduler on dynamic start_date
41 Using a dynamic start_date (e.g. `start_date = datetime.now()`) is not considered a best practice. The 1.8.0 scheduler
42 is less forgiving in this area. If you encounter DAGs not being scheduled you can try using a fixed start_date and
43 renaming your dag. The last step is required to make sure you start with a clean slate, otherwise the old schedule can
44 interfere.

46 ### New and updated scheduler options
47 Please read through these options, defaults have changed since 1.7.1.
48 
49 #### child_process_log_directory
50 In order the increase the robustness of the scheduler, DAGS our now processed in their own process. Therefore each
51 DAG has its own log file for the scheduler. These are placed in `child_process_log_directory` which defaults to
52 `<AIRFLOW_HOME>/scheduler/latest`. You will need to make sure these log files are removed.
53 
54 > DAG logs or processor logs ignore and command line settings for log file locations.
55 
56 #### run_duration
57 Previously the command line option `num_runs` was used to let the scheduler terminate after a certain amount of
58 loops. This is now time bound and defaults to `-1`, which means run continuously. See also num_runs.
59 
60 #### num_runs
61 Previously `num_runs` was used to let the scheduler terminate after a certain amount of loops. Now num_runs specifies
62 the number of times to try to schedule each DAG file within `run_duration` time. Defaults to `-1`, which means try
63 indefinitely. This is only available on the command line.
64 
65 #### min_file_process_interval
66 After how much time should an updated DAG be picked up from the filesystem.
67 
68 #### dag_dir_list_interval
69 How often the scheduler should relist the contents of the DAG directory. If you experience that while developing your
70 dags are not being picked up, have a look at this number and decrease it when necessary.
71 
72 #### catchup_by_default
73 By default the scheduler will fill any missing interval DAG Runs between the last execution date and the current date.
74 This setting changes that behavior to only execute the latest interval. This can also be specified per DAG as
75 `catchup = False / True`. Command line backfills will still work.
76 
77 ### Faulty Dags do not show an error in the Web UI
78 
79 Due to changes in the way Airflow processes DAGs the Web UI does not show an error when processing a faulty DAG. To
80 find processing errors go the `child_process_log_directory` which defaults to `<AIRFLOW_HOME>/scheduler/latest`.
81 
82 ### New DAGs are paused by default
83 
84 Previously, new DAGs would be scheduled immediately. To retain the old behavior, add this to airflow.cfg:
85 
86 ```
87 [core]
88 dags_are_paused_at_creation = False
89 ```

91 ### Airflow Context variable are passed to Hive config if conf is specified
92 
93 If you specify a hive conf to the run_cli command of the HiveHook, Airflow add some
94 convenience variables to the config. In case your run a sceure Hadoop setup it might be
95 required to whitelist these variables by adding the following to your configuration:
96 
97 ```
98 <property>
99  <name>hive.security.authorization.sqlstd.confwhitelist.append</name>
100  <value>airflow\.ctx\..*</value>
101 </property>
102 ```
103 ### Google Cloud Operator and Hook alignment
104 
105 All Google Cloud Operators and Hooks are aligned and use the same client library. Now you have a single connection
106 type for all kinds of Google Cloud Operators.
107 
108 If you experience problems connecting with your operator make sure you set the connection type "Google Cloud Platform".
109 
110 Also the old P12 key file type is not supported anymore and only the new JSON key files are supported as a service
111 account.

113 ### Deprecated Features
114 These features are marked for deprecation. They may still work (and raise a `DeprecationWarning`), but are no longer
115 supported and will be removed entirely in Airflow 2.0
116 
117 - Hooks and operators must be imported from their respective submodules
118 
119  `airflow.operators.PigOperator` is no longer supported; `from airflow.operators.pig_operator import PigOperator` is.
120  (AIRFLOW-31, AIRFLOW-200)
121 
122 - Operators no longer accept arbitrary arguments
123 
124  Previously, `Operator.__init__()` accepted any arguments (either positional `*args` or keyword `**kwargs`) without
125  complaint. Now, invalid arguments will be rejected. (https://github.com/apache/incubator-airflow/pull/1285)

127 ### Known Issues
128 There is a report that the default of "-1" for num_runs creates an issue where errors are reported while parsing tasks.
129 It was not confirmed, but a workaround was found by changing the default back to `None`.
130 
131 To do this edit `cli.py`, find the following:
132 
133 ```
134  'num_runs': Arg(
135  ("-n", "--num_runs"),
136  default=-1, type=int,
137  help="Set the number of runs to execute before exiting"),
138 ```
139 
140 and change `default=-1` to `default=None`. Please report on the mailing list if you have this issue.
